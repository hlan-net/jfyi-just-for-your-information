# Multi-Tenant Cloud Architecture (OAuth 2.1 + RBAC)

**Roadmap phase:** 1 — Foundation (Accelerated)
**Status:** In Progress

## Problem

JFYI currently assumes a single operator: the developer who deploys it owns everything it stores and can call every tool and endpoint. This model breaks down in team or enterprise deployments where multiple developers share a JFYI instance and need differentiated access.

## Proposed Solution

Transform JFYI from a single-tenant local tool into a multi-tenant, cloud-ready SaaS architecture.

Key changes:
1. **Vue 3 + Vite SPA**: The dashboard will be rewritten from vanilla HTML to a modern Vue 3 SPA for a richer user experience, including conditional rendering for settings and admin dashboards.
2. **Multiple Identity Providers (IdPs)**: Native integration with GitHub, Google, and Entra ID via `authlib`.
3. **Account Linking**: Users can log in using any supported IdP, and their identities are bound to a single internal `user_id`.
4. **Dual Authentication Strategy**:
   - **Dashboard**: HTTP-only signed session cookies for secure browser interactions.
   - **MCP Client**: Long-lived JSON Web Tokens (JWTs) generated by the user in the dashboard UI and passed via `Authorization: Bearer` by MCP clients (e.g., Claude Desktop) to `dispatch_tool` and SSE endpoints.
5. **Admin Role**: The first user created in the database is automatically granted `is_admin = True`. Admins can perform CRUD operations on other users' accounts and linked identities.

## Implementation Phases

### Phase 1: Database Schema Rewrite
Drop the existing legacy tables and introduce a tenant-scoped schema:
- `users`: Core account (id, name, email, is_admin).
- `user_identities`: Provider links (user_id, provider, sub).
- `profile_rules`, `agents`, `interactions`, `friction_events`: Data tables updated with `user_id` to strictly silo data per tenant.

### Phase 2: Authentication & JWT Core
Add `authlib`, `itsdangerous`, and `pyjwt`. Configure the OAuth registry for IdPs and implement JWT encoding/decoding utilities for MCP API keys.

### Phase 3: Backend API Restructuring
Add `SessionMiddleware` for OAuth flows (`/auth/login/{provider}`, `/auth/callback/{provider}`). Implement Fastapi dependencies `get_current_user` and `get_admin_user` to secure routes. Expose `/api/keys` for users to generate MCP API keys.

### Phase 4: MCP Server Authentication
Protect the `/mcp/sse` and `/mcp/messages/` endpoints. Pass the authenticated `user_id` into the `dispatch_tool` execution context.

### Phase 5: Frontend SPA (Vue 3 + Vite)
Build a modular frontend with `vue-router` covering views for login, dashboard, developer profile, account settings, and admin user management. Configure Vite to bundle the static output for FastAPI mounting.

## Security

- All API routes and tool endpoints require explicit contextual authorization.
- Token and Session invalidation procedures ensure revoked identity links immediately terminate access.
- Cross-Site Request Forgery (CSRF) protection via `SessionMiddleware`.
